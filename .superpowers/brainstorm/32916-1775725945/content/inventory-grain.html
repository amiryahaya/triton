Crypto Inventory Table — Row Grain

What does one row represent?

The inventory view aggregates findings across every machine in the org. The key question is how finely to group them — too coarse and you can't see which key sizes are the problem; too fine and the table becomes noisy.

A

One row per (algorithm, key_size) — flat

Each distinct algorithm+size combination is its own row. Natural fit for PQC classification (which varies by size).

Row counts (realistic org): ~20–40 rows across all categories. Sortable by any column. Click a row to drill down to findings filtered by that (algo, size).

B

Grouped by algorithm, expandable by size

Each algorithm family collapses into one row that expands on click to reveal per-size details.

Hybrid — starting point depends on what you pick

Both options can support the same drill-down (click → filtered findings list). Both can support filters/sorts. The difference is really whether an operator's first glance should show "RSA-2048 is the one killing me" (Option A) or "RSA is the family killing me, let me expand to see which size" (Option B).
